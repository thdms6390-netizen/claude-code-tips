MCP 툴 결과 최대 500K — DB 스키마 통째로 넘기기

MCP 툴 응답이 너무 길어서 잘리던 문제, 이제 500K까지 늘릴 수 있음. DB 스키마 같은 덩어리 큰 데이터도 그대로 전달 가능.

NEWMCP2026.4.3 · v2.1.91
›

이게 뭔데?

기존엔 MCP 툴이 반환하는 결과 크기에 제한이 있어서, DB 스키마나 긴 파일 목록 같은 데이터가 중간에 잘렸음. 이제 MCP 툴 응답의 _meta에 anthropic/maxResultSizeChars 어노테이션을 넣으면 최대 500K까지 늘릴 수 있음.

어떻게 써?

MCP 서버 쪽 툴 응답에 아래처럼 메타 정보를 추가하면 됨:

# MCP 툴 result에 _meta 추가 (서버 코드 예시)
{
  "content": "...긴 DB 스키마...",
  "_meta": {
    "anthropic/maxResultSizeChars": 500000
  }
}

이런 경우에 유용함

  • DB 전체 스키마를 한 번에 Claude에게 넘길 때
  • 대형 JSON 설정 파일을 MCP로 읽어올 때
  • 툴 결과가 잘려서 Claude가 엉뚱한 답을 냈던 경우

--resume 개선 — 자동화 세션은 목록에서 빠짐

--resume으로 세션 고를 때 claude -p나 SDK로 만들어진 세션은 이제 안 보여. 내가 직접 쓴 세션만 딱 나옴.

NEWUX 개선실용적2026. 4. 2. · v2.1.90
›

뭐가 바뀐 거야?

전에는 --resume 피커를 열면 자동화 스크립트나 SDK가 만든 세션까지 전부 섞여서 나왔어. 이제는 그런 세션들은 걸러지고, 내가 직접 대화한 세션만 보여줌.

  • claude -p (파이프 모드)로 생성된 세션 → 목록에서 제외
  • SDK 호출로 생성된 세션 → 목록에서 제외
  • 덕분에 이어하고 싶은 세션을 훨씬 빠르게 찾을 수 있음

사용법

# 이전 세션 목록에서 골라서 이어하기
claude --resume

# 특정 세션 ID로 바로 이어하기
claude --resume 세션ID

Edit 툴 개선 — Read 없이 바로 편집 가능

sed -n이나 cat으로 본 파일을 별도 Read 호출 없이 바로 Edit할 수 있게 됨. 불필요한 단계가 줄어서 더 빠르게 작업 가능.

NEW편집실용적2026. 4. 1. · v2.1.89
›

뭐가 바뀐 거야?

기존엔 Bash로 cat이나 sed -n으로 파일 내용을 봤더라도, Edit 툴을 쓰려면 다시 Read를 해야 했음. 이제는 그 과정이 필요 없어. Bash로 파일 내용을 확인한 다음 곧바로 Edit 툴로 수정할 수 있는 거야.

전/후 비교

# 기존: Bash로 보고 → Read로 다시 읽고 → Edit
Bash: sed -n '10,20p' src/index.ts
Read: src/index.ts   ← 이 단계가 필요했음
Edit: src/index.ts ...

# 이제: Bash로 보고 → 바로 Edit
Bash: sed -n '10,20p' src/index.ts
Edit: src/index.ts ...  ← 바로 가능!

MCP 연결 속도 개선 — 느린 서버 때문에 멈추지 않음

헤드리스 모드(-p)에서 MCP 연결을 기다리지 않고 바로 실행할 수 있음. 서버 연결은 최대 5초만 기다리도록 바뀜.

NEWMCP자동화2026. 4. 1. · v2.1.89
›

뭐가 달라진 거야?

  • MCP_CONNECTION_NONBLOCKING=true — -p 모드에서 MCP 연결 대기를 완전히 건너뜀. CI/자동화 환경에서 느린 MCP 서버 때문에 파이프라인이 멈추는 일이 없어짐.
  • 연결 타임아웃 5초 제한 — --mcp-config로 지정한 서버 연결이 이제 최대 5초만 기다림. 가장 느린 서버 때문에 전체가 블로킹되는 상황이 없어지는 거야.

사용법

# MCP 연결 대기 완전히 스킵 (자동화/CI 환경에서 유용)
MCP_CONNECTION_NONBLOCKING=true claude -p "작업 내용"

# --mcp-config 사용 시 느린 서버는 5초 후 자동 스킵됨
claude --mcp-config ./mcp.json -p "작업 내용"

서브에이전트 @멘션 — 이름으로 바로 불러오기

이제 @멘션 자동완성에 이름 붙은 서브에이전트가 뜸. 여러 에이전트 굴릴 때 훨씬 편해짐.

NEW멀티에이전트자주쓰임2025.3.31 · v2.1.88
›

이게 뭔데?

Claude Code에서 서브에이전트(subagent)를 여러 개 띄워 쓸 때, 각각에 이름을 붙여두면 이제 @ 타이핑만 해도 자동완성 목록에 바로 뜸. 에이전트 이름 기억해서 직접 입력할 필요 없이 골라서 멘션하면 끝.

이런 식으로 써봐

# 프롬프트창에서 @만 치면 등록된 서브에이전트 목록이 뜸
@reviewer 이 코드 리뷰해줘

# 이름 일부만 쳐도 필터링됨
@test  → test-runner, tester 등 자동완성 후보 등장

왜 좋냐면

  • 에이전트 이름 외울 필요 없이 @ 하나로 목록 탐색 가능
  • 멀티에이전트 워크플로우 짤 때 실수 줄어듦
  • 팀에서 공유하는 에이전트 구성이 있을 때 특히 편함

PermissionDenied 훅 — 거절당했을 때 재시도 제어하기

자동 모드에서 권한 거절이 뜰 때 훅으로 가로채서 "다시 해봐"라고 모델에게 알려줄 수 있음.

NEW훅/자동화자동화2025.3.31 · v2.1.88
›

어떻게 동작하냐면

Claude Code의 auto 모드 classifier가 특정 작업을 거절(deny)했을 때, 이제 PermissionDenied 훅이 발동됨. 이 훅에서 {retry: true}를 반환하면 모델한테 "이거 다시 시도해도 됨"이라고 신호를 보낼 수 있는 거야.

훅 설정 예시

# .claude/hooks.js (예시)
export const PermissionDenied = async ({ tool, reason }) => {
  // 특정 조건일 때만 재시도 허용
  if (tool === 'read_file' && reason === 'sandbox') {
    return { retry: true };
  }
  // 기본은 그냥 거절 유지
  return {};
};

이럴 때 유용해

  • 자동화 파이프라인에서 일시적 권한 오류를 훅으로 핸들링하고 싶을 때
  • 거절 사유에 따라 재시도 여부를 세밀하게 컨트롤하고 싶을 때
  • 모델이 거절에 막혀 멈추는 대신 다른 방법을 스스로 찾게 하고 싶을 때

Hooks 조건부 실행 — 원하는 상황에서만 훅 발동

훅에 if 조건을 달아서 특정 명령어에서만 실행되게 할 수 있음. 불필요한 프로세스 낭비 없이 훅을 정교하게 제어 가능.

NEWHooks실용적자주쓰임2026.03.27 · v2.1.85
›

이게 왜 좋냐면

기존엔 훅이 모든 상황에서 무조건 실행됐는데, 이제 if 필드로 조건을 걸 수 있음. 예를 들어 git 관련 명령어를 실행할 때만 훅이 발동되게 하거나, 특정 파일 작업에서만 동작하게 제한할 수 있는 거야. 쓸데없이 훅이 자꾸 뜨는 거 방지됨.

설정 예시 (settings.json)

# git 명령어 실행할 때만 이 훅이 발동됨
{
  "hooks": {
    "PreToolUse": [
      {
        "if": "Bash(git *)",
        "command": "echo 'git 명령어 감지!'"
      }
    ]
  }
}

조건 문법

권한 규칙 문법을 그대로 사용함. Bash(git *), Write(src/**) 같은 식으로 툴 이름 + 패턴으로 쓰면 됨.

MCP headersHelper 멀티 서버 지원 — 하나의 헬퍼로 여러 서버 처리

MCP 헤더 헬퍼 스크립트에서 어느 서버에 연결 중인지 알 수 있게 됨. 서버마다 헬퍼를 따로 안 만들어도 돼.

NEWMCP설정2026.03.27 · v2.1.85
›

추가된 환경변수

  • CLAUDE_CODE_MCP_SERVER_NAME — 현재 연결 중인 MCP 서버 이름
  • CLAUDE_CODE_MCP_SERVER_URL — 현재 연결 중인 MCP 서버 URL

활용 예시 (headersHelper 스크립트)

# 서버 이름에 따라 다른 토큰을 돌려줌
if [ "$CLAUDE_CODE_MCP_SERVER_NAME" = "my-api-server" ]; then
  echo '{"Authorization": "Bearer token-for-api"}'
else
  echo '{"Authorization": "Bearer token-default"}'
fi

MCP 서버 여러 개 운영 중이면 이제 헬퍼 하나로 다 처리할 수 있는 거야.

Computer Use — Claude가 직접 마우스 움직이고 클릭하는 거

진짜로 화면 보고 → 마우스 이동 → 클릭 → 타이핑까지. 사람이 쓰듯이 앱을 조작.

NEW2026.03.24 · 별도 제품 발표
›

이게 뭔데?

Claude가 화면을 스크린샷으로 계속 캡처하면서, 어디를 클릭하고 뭘 타이핑할지 판단합니다. 말 그대로 사람이 컴퓨터 쓰는 것처럼 앱을 조작합니다.

작동 우선순위 (3단계)

  • 1순위: Connector — Gmail, Slack, Calendar 등 API가 있으면 직접 연결 (가장 빠름)
  • 2순위: 브라우저 — API 없으면 Chrome 열어서 웹 UI 조작
  • 3순위: 화면 제어 — 둘 다 안 되면 마우스+키보드로 직접 조작 (이게 마우스 움직이는 거)

실제로 할 수 있는 것들

  • Mac 앱 열어서 버튼 클릭, 폼 입력, 메뉴 탐색
  • 폰 시뮬레이터 열어서 개발한 앱 UX 테스트
  • 여러 소스에서 데이터 모아서 스프레드시트 채우기
  • API 없는 내부 대시보드/레거시 툴 조작
  • IDE에서 코드 수정 → 테스트 실행 → PR 제출

Channels — 외부에서 Claude 세션에 메시지를 "보내는" 기능

Telegram, Discord, CI 알림 등이 터미널로 바로 들어옴. Claude가 즉시 반응.

NEW실시간2026.03.21 · v2.1.81
›

기존 방식과 뭐가 다른데?

  • MCP = Claude가 필요할 때 "가져오는" 것 (pull)
  • Channels = 외부에서 Claude한테 "밀어넣는" 것 (push)

예를 들어 CI 빌드가 실패하면 → 알림이 터미널에 바로 뜨고 → Claude가 에러 로그를 읽고 → 알아서 디버깅 시작합니다.

실제 사용 예시: Telegram에서 코딩 지시하기

커피 마시면서 폰으로 Telegram 메시지를 보내면, 내 컴퓨터에서 돌고 있는 Claude가 코드를 수정합니다.

# 1. Telegram 봇 만들기 (BotFather에서)
# 2. 플러그인 설치
/plugin install telegram@claude-plugins-official

# 3. 토큰 설정
/telegram:configure <봇-토큰>

# 4. Channels 켜고 시작
claude --channels plugin:telegram@claude-plugins-official

# 5. Telegram에서 봇에게 메시지 → 페어링 코드 받기
/telegram:access pair <코드>
/telegram:access policy allowlist  # 나만 허용
💡 Channels는 세션이 켜져 있을 때만 작동합니다. 24/7 자동화가 필요하면 Scheduled Tasks를 쓰세요.

/btw — 작업 중에 딴 질문 하기 (컨텍스트 안 더러워짐)

Claude가 코딩하는 중에 궁금한 거 물어봐도 메인 대화에 안 섞임. 토큰도 절약.

NEW자주쓰임2026.03.10 · v2.1.72
›

이게 왜 중요한데?

Claude가 열심히 코드 작성 중일 때 "아 이 라이브러리 최신 버전 뭐지?" 같은 질문을 하면, 그게 대화 기록에 쌓여서 컨텍스트가 낭비됩니다. /btw는 이걸 해결합니다:

  • 질문과 답변이 대화 기록에 안 남음 → 토큰 절약 (최대 ~50%)
  • 메인 작업이 계속 돌아감 (백그라운드에서 중단 안 됨)
  • 현재 대화 컨텍스트를 다 볼 수 있음 (Claude가 읽은 코드 참조 가능)

사용법

# Claude가 코딩 중일 때 그냥 입력:
/btw 이 프로젝트에서 쓰는 ORM이 뭐야?
/btw React 19에서 useEffect 변경된 거 있어?
/btw 아까 수정한 함수 왜 그렇게 짠 거야?

제약사항

  • 읽기 전용 — 파일 수정이나 명령어 실행은 불가
  • 1회성 — 한 질문에 한 답변, 후속 대화 없음
  • 프롬프트 캐시를 재사용해서 추가 비용 최소
💡 긴 세션에서 잡다한 질문이 많을수록 효과 큼. "이거 뭐지?" 싶으면 무조건 /btw로 물어보세요.
