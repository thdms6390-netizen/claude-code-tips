뭐가 달라졌냐

Dynamic Workflow Size + /code-review 분리

워크플로 에이전트 규모를 직접 조절하고, 코드 리뷰도 빠른 버전과 멀티에이전트 버전으로 나뉨.

NEW자동화자주쓰임2026. 7. 7. · v2.1.202
›

워크플로 에이전트 수 조절

/config에 Dynamic workflow size 설정이 추가됨. 워크플로가 내부적으로 몇 개의 에이전트를 생성할지 small / medium / large 세 단계로 조절할 수 있어. 하드 캡이 아니라 "이 정도 규모로 해줘" 하는 권고 수준이라서 실제로는 작업 복잡도에 따라 유연하게 적용됨.

/review vs /code-review 분리

이전에 /review가 멀티 에이전트로 바뀐 적 있었는데, 이번에 다시 빠른 단일 패스 리뷰로 되돌아왔어. 무거운 멀티 에이전트 리뷰가 필요하면 새 명령어를 써야 함.

  • /review <pr> — 빠른 단일 패스 리뷰 (가볍게 쓸 때)
  • /code-review <level> <pr#> — 에포트 레벨 지정 멀티 에이전트 리뷰 (깊게 팔 때)

사용법

# 워크플로 에이전트 규모 설정 (small / medium / large)
/config Dynamic workflow size medium

# 빠른 PR 리뷰 (단일 패스)
/review 123

# 멀티 에이전트 리뷰 (에포트 레벨 선택 가능)
/code-review high 123

기본 권한 모드 → Manual + 다이얼로그 자동계속 제거

이제 Claude가 멋대로 이어서 진행하지 않음. 물어보면 진짜로 기다림.

NEW자동화자주쓰임2026.07.04 · v2.1.200
›

뭐가 바뀐 거야?

  • AskUserQuestion 다이얼로그가 더 이상 자동으로 이어서 실행되지 않음 — 이전엔 타임아웃 지나면 그냥 계속 진행했는데, 이제 진짜로 멈추고 기다림
  • 기존 default 권한 모드가 공식적으로 Manual로 이름 변경 — CLI, --help, VS Code, JetBrains 전반에 적용
  • 자동으로 진행시키고 싶으면 /config에서 idle timeout을 직접 켜야 함

왜 좋은 거야?

백그라운드에서 Claude가 질문 던지고 아무도 답 안 했는데 혼자 판단해서 계속 진행하는 상황이 방지됨. 특히 자동화 파이프라인이나 서버리스 환경에서 의도치 않은 실행 사고를 막을 수 있음.

권한 모드 설정법

# CLI 플래그로 지정 (manual이 이제 공식 이름)
claude --permission-mode manual

# 설정 파일에서 지정
"defaultMode": "manual"

# idle timeout 켜기 (다이얼로그 자동 계속 원할 때만)
/config askUserQuestion.idleTimeout=30

백그라운드 에이전트 알림 — 입력 필요하거나 끝나면 훅 발동

claude agents 뷰에서 백그라운드 세션이 입력을 기다리거나 완료되면 Notification 훅이 자동으로 발동됨. 폰 알림이나 외부 연동과 연결하면 진짜 편해짐.

NEW자동화자주쓰임2026.7.2 · v2.1.198
›

어떻게 동작하냐면

  • agent_needs_input — 백그라운드 세션이 승인이나 입력을 기다릴 때 발동
  • agent_completed — 백그라운드 세션이 작업을 끝냈을 때 발동
  • 기존 Notification 훅 인프라를 그대로 쓰기 때문에 이미 훅 설정해 둔 사람은 바로 동작함

settings.json에 훅 등록하는 방법

# ~/.claude/settings.json
{
  "hooks": {
    "Notification": [
      {
        "matcher": "agent_needs_input",
        "command": "notify-send '클로드가 입력을 기다려요'"
      },
      {
        "matcher": "agent_completed",
        "command": "notify-send '백그라운드 작업 완료!'"
      }
    ]
  }
}

실제 활용 예시

  • 백그라운드에서 테스트 돌리고, 끝나면 슬랙 DM으로 결과 받기
  • 긴 작업 켜두고 자리 비웠다가 폰 알림 받으면 돌아오기
  • /goal이나 claude agents로 여러 세션 돌릴 때 특히 유용함

세션 자동 이름

예전엔 세션 이름이 무미건조한 ID나 타임스탬프였는데, 이제 시작할 때부터 내용을 반영한 읽기 좋은 이름이 자동으로 붙음. 여러 세션 관리할 때 "이게 뭐 하던 세션이었더라?" 헷갈리는 일이 줄어듦.

파일 첨부 클릭으로 열기

채팅창에 올라온 파일 첨부를 Cmd(Mac) 또는 Ctrl(Windows/Linux)-클릭하면 해당 파일이 Finder 또는 파일 탐색기에서 바로 강조 표시됨. 어느 경로인지 찾으러 터미널 뒤질 필요 없음.

에이전트 뷰 전환 개선

포그라운드 세션에서 에이전트 뷰 열 때 기존엔 ←를 두 번 눌러야 했는데, 이제 한 번만 누르면 됨. 백그라운드 세션과 동일한 동작으로 통일.

# 파일 첨부 열기
Cmd + 클릭  ← Mac
Ctrl + 클릭 ← Windows / Linux

# 에이전트 뷰 전환 (포그라운드 세션)
← ← 한 번만 누르면 됨 (기존엔 두 번)

  • 아직 검증 안 된 최신 버전 쓰는 걸 막고 싶을 때

  "requiredMaximumVersion": "2.1.163"
}

범위 밖 버전으로 실행하면 Claude Code가 시작을 거부하고 승인된 버전으로 안내하는 메시지를 띄워줌.

셸 시작 파일 쓰기 전 확인 프롬프트 — 의도치 않은 실행 방지

Claude가 .zshenv, .zlogin, .bash_login, ~/.config/git/에 쓰려 할 때 이제 먼저 물어봄. 자동으로 셸 설정이 바뀌는 사고를 막아주는 거.

NEW보안2026.6.2 · v2.1.160

왜 중요해?

셸 시작 파일(.zshenv, .zlogin, .bash_login)이나 Git 설정 디렉터리(~/.config/git/)는 터미널을 열 때마다 자동 실행되는 스크립트가 담겨 있어. Claude가 여기에 뭔가를 몰래 쓰면 의도치 않은 명령이 계속 실행될 수 있었는데, 이제 반드시 확인 단계를 거치게 됨.

  • Claude가 해당 파일에 쓰기를 시도하면 확인 프롬프트가 뜸
  • 사용자가 명시적으로 허용해야만 실제로 쓰기가 진행됨
  • Auto Mode에서도 이 확인은 생략되지 않음

claude plugin init — 플러그인 뼈대 한 줄로 생성

새 플러그인 만들 때 디렉토리 구조·파일 자동 생성. 처음부터 안 짜도 됨.

NEW플러그인실용적2026.05.30 · v2.1.157
›

이게 뭔데?

claude plugin init <name>을 실행하면 .claude/skills/ 안에 플러그인 뼈대를 자동으로 만들어줌. 파일 구조, 기본 설정 파일까지 한 번에 생성되니까 처음 플러그인 만드는 사람도 헤맬 필요 없음.

같이 추가된 것들

  • /plugin 자동완성 — 서브커맨드, 설치된 플러그인 이름, 마켓플레이스 플러그인까지 탭 키로 자동완성
  • 워크플로우 키워드 트리거 설정 — /config에서 "Workflow keyword trigger" 켜고 끄기 가능. 프롬프트에 "workflow"라는 단어가 들어가도 다이나믹 워크플로우가 자동 발동되지 않게 차단할 수 있음

사용법

# 새 플러그인 스캐폴드 생성
claude plugin init my-plugin

# 생성되는 경로
.claude/skills/my-plugin/

# /plugin 서브커맨드 자동완성 (탭 누르면 목록 뜸)
/plugin [Tab]

# 워크플로우 키워드 트리거 끄기
/config → Workflow keyword trigger → off

skipLfs — LFS 파일 빼고 플러그인 클론하기

github/git 플러그인 소스에서 Git LFS 다운로드를 건너뛸 수 있는 옵션. 무거운 바이너리 포함된 플러그인 설치할 때 속도 확 줄어듦.

NEW플러그인2026.05.28 · v2.1.153
›

이게 왜 필요해?

플러그인 소스가 Git LFS로 대용량 파일(모델, 바이너리, 에셋 등)을 관리하는 경우, 설치할 때 그 파일까지 전부 받아오느라 느려지는 문제가 있었어. skipLfs: true를 설정하면 LFS 포인터 파일만 받고 실제 파일은 스킵해서 설치가 훨씬 빨라짐.

사용법

# CLAUDE.json 또는 플러그인 설정 파일에서
{
  "plugins": [
    {
      "source": "github",
      "repo": "some-org/some-plugin",
      "skipLfs": true
    }
  ]
}

언제 쓰면 좋아?

  • 플러그인 설치 속도가 비정상적으로 느릴 때
  • LFS 파일이 실제 플러그인 동작에 필요 없는 경우 (문서, 테스트용 에셋 등)
  • CI/CD 환경에서 플러그인 설치 시간 줄이고 싶을 때

worktree.bgIsolation: "none" — 워크트리 없이 백그라운드 세션 돌리기

워크트리 쓰기 어려운 환경에서, 백그라운드 세션이 현재 작업 디렉토리를 그냥 직접 편집하게 할 수 있음.

›

언제 쓰냐면

모노레포, 대형 레포, 특수한 빌드 환경 등에서 Git worktree 자체가 불가능하거나 불편한 경우가 있음. 기존엔 백그라운드 세션이 무조건 EnterWorktree를 거쳐야 했는데, 이제 bgIsolation: "none" 설정 하나로 그 과정 없이 작업 디렉토리를 바로 편집하게 할 수 있음.

설정 방법

{
  "worktree": {
    "bgIsolation": "none"
  }
}

백그라운드 세션이 현재 브랜치 파일을 직접 건드리게 되므로, 충돌 위험이 있는 환경에서는 신중하게 써야 함. worktree가 실용적인 환경이라면 기본 격리 방식을 유지하는 게 안전함.

claude agents 플래그 대폭 확장 — 백그라운드 세션 세밀하게 제어하기

디스패치된 백그라운드 에이전트 세션에 모델, 에포트, MCP 설정, 권한 모드까지 직접 지정할 수 있게 됨.

NEW에이전트자동화2026.5.15 · v2.1.142
›

추가된 플래그 목록

  • --model — 백그라운드 세션에서 쓸 모델 지정
  • --effort — 에포트 수준 직접 설정 (low / medium / high / xhigh)
  • --permission-mode — 권한 모드 선택 (auto, bypassPermissions 등)
  • --mcp-config — 별도 MCP 설정 파일 경로 지정
  • --settings — 커스텀 설정 파일 적용
  • --plugin-dir — 플러그인 디렉터리 경로 지정
  • --add-dir — 세션에 추가 작업 디렉터리 포함
  • --dangerously-skip-permissions — 권한 확인 완전 생략 (위험, 신중히)

사용 예시

# 특정 모델 + 에포트로 백그라운드 에이전트 실행
claude agents --model claude-opus-4-5 --effort high

# MCP 설정 + 플러그인 디렉터리 따로 지정
claude agents --mcp-config ./mcp-bg.json --plugin-dir ./plugins

# 자동화 파이프라인용: 권한 확인 생략 + auto 모드
claude agents --permission-mode auto --dangerously-skip-permissions

이게 왜 좋냐면

이전까지는 claude agents로 백그라운드 세션을 띄울 때 설정을 세밀하게 제어하기가 어려웠는데, 이제 메인 세션이랑 완전히 다른 모델·권한·MCP 구성으로 에이전트를 돌릴 수 있음. 예를 들어 무거운 작업엔 Opus + xhigh, 빠른 반복 작업엔 Sonnet + low 이런 식으로 분리해서 운용 가능.

이게 왜 필요해?

  }
}

      }
    ]
  }
}

# 또는 JSON으로 블록

PermissionDenied 훅 — 거절당했을 때 재시도 제어하기

자동 모드에서 권한 거절이 뜰 때 훅으로 가로채서 "다시 해봐"라고 모델에게 알려줄 수 있음.

NEW훅/자동화자동화2025.3.31 · v2.1.88
›

어떻게 동작하냐면

Claude Code의 auto 모드 classifier가 특정 작업을 거절(deny)했을 때, 이제 PermissionDenied 훅이 발동됨. 이 훅에서 {retry: true}를 반환하면 모델한테 "이거 다시 시도해도 됨"이라고 신호를 보낼 수 있는 거야.

훅 설정 예시

# .claude/hooks.js (예시)
export const PermissionDenied = async ({ tool, reason }) => {
  // 특정 조건일 때만 재시도 허용
  if (tool === 'read_file' && reason === 'sandbox') {
    return { retry: true };
  }
  // 기본은 그냥 거절 유지
  return {};
};

이럴 때 유용해

  • 자동화 파이프라인에서 일시적 권한 오류를 훅으로 핸들링하고 싶을 때
  • 거절 사유에 따라 재시도 여부를 세밀하게 컨트롤하고 싶을 때
  • 모델이 거절에 막혀 멈추는 대신 다른 방법을 스스로 찾게 하고 싶을 때

}

/btw — 작업 중에 딴 질문 하기 (컨텍스트 안 더러워짐)

Claude가 코딩하는 중에 궁금한 거 물어봐도 메인 대화에 안 섞임. 토큰도 절약.

NEW자주쓰임2026.03.10 · v2.1.72
›

이게 왜 중요한데?

Claude가 열심히 코드 작성 중일 때 "아 이 라이브러리 최신 버전 뭐지?" 같은 질문을 하면, 그게 대화 기록에 쌓여서 컨텍스트가 낭비됩니다. /btw는 이걸 해결합니다:

  • 질문과 답변이 대화 기록에 안 남음 → 토큰 절약 (최대 ~50%)
  • 메인 작업이 계속 돌아감 (백그라운드에서 중단 안 됨)
  • 현재 대화 컨텍스트를 다 볼 수 있음 (Claude가 읽은 코드 참조 가능)

사용법

# Claude가 코딩 중일 때 그냥 입력:
/btw 이 프로젝트에서 쓰는 ORM이 뭐야?
/btw React 19에서 useEffect 변경된 거 있어?
/btw 아까 수정한 함수 왜 그렇게 짠 거야?

제약사항

  • 읽기 전용 — 파일 수정이나 명령어 실행은 불가
  • 1회성 — 한 질문에 한 답변, 후속 대화 없음
  • 프롬프트 캐시를 재사용해서 추가 비용 최소
💡 긴 세션에서 잡다한 질문이 많을수록 효과 큼. "이거 뭐지?" 싶으면 무조건 /btw로 물어보세요.
